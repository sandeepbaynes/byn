# Glossary

Terms used throughout the docs and code.

---

### AAD — Additional Authenticated Data

In AEAD (XChaCha20-Poly1305), the AAD is a byte string that is NOT
encrypted but IS covered by the auth tag. Any change to the AAD makes
decryption fail.

byn uses AAD to bind ciphertext to context:
- **Row values:** AAD = `vault_id ‖ 0x1F ‖ kind ‖ 0x1F ‖ name`. Means
  a row literally copied between vaults or onto another entry's slot
  fails to decrypt.
- **Wrapped key:** AAD = full header bytes (version + salt + Argon2
  params + nonce). Means tampering with the header (e.g., downgrading
  Argon2 params) fails to unwrap.

### AEAD

Authenticated Encryption with Associated Data. The primitive family
that gives you both confidentiality (encryption) and integrity (a
tag verifying the message + AAD haven't been touched). byn uses
XChaCha20-Poly1305 throughout.

### Agent mode

When `--json` appears in the CLI args (before any `--`), byn:
- Emits machine-readable output where applicable
- **Refuses to prompt** for anything (no `trust [y/N]:`)
- Hard-fails on untrusted `.byn` instead of asking
- Hard-fails on tampered `.byn`

Lets CI, scripts, and AI coding agents interact with
byn safely.

### Argon2id

The memory-hard password-hashing function we use to derive a vault
key's wrapping key from the master password.

Default parameters (`DefaultArgon2Params`): time=2, memory=64 MiB,
threads=4, key length=32 — tuned for ~1s on a laptop. Bounds: time ∈
[1, 16], memory ∈ [8 MiB, 1 GiB], threads ∈ [1, 16] — the upper limits
prevent DoS via malicious headers, the lower limits reject trivial
attacker-supplied params.

### Audit chain

A sequence of HMAC-SHA256 hashes, where each event's hash depends on
the previous event's. Any insertion, deletion, or modification
breaks the chain at the first affected event.

Seed: 32 random bytes stored in the vault meta. Head (latest hash):
also in meta, so the daemon can resume cleanly on cold start.

### Daemon

The long-running background process. Owns the vault key (only while
unlocked), the SQLite handles, the audit logger, and the socket
listener. Started as `byn daemon start`.

### Discovery

The walk from CWD upward looking for a `.byn` file. See
[`.byn` file format](byn-file-format.md).

### Entry

A single key→value pair within a `(vault, project, env)` scope. The
canonical secret unit. Today: env-vars. Phase 5 adds files.

### Env (within a vault)

The leaf level of the scope hierarchy. Examples: `default`, `dev`,
`staging`, `prod`.

Non-default envs **inherit** from the project's default env: if
`STAGING.DB_URL` doesn't exist, `byn get DB_URL --env staging`
falls back to `DEFAULT.DB_URL`.

### Fingerprint (machine fingerprint)

A SHA-256 of stable hardware identifiers (CPU ID, board serial,
primary MAC). Used to bind the wrapped vault key to the local
machine — a copy of `wrapped.key` won't unwrap elsewhere.

Made portable in Phase 6 for cloud sync (the cloud variant strips the
fingerprint binding).

### Fingerprint (wrapped-key fingerprint, in `meta.json`)

SHA-256 of the `wrapped.key` file bytes. The daemon checks this on
every vault open. Catches an attacker who swaps the wrapped key for
one they control (e.g., from another vault they unlocked).

### Hint

A short stderr echo after a mutating op. `Stored "DB_URL" in
default/billing/staging.` Gated by `BYN_HINTS` and TTY status.
Written via `cmd/byn/hints.go:hintf`.

### IPC

Inter-process communication. The daemon ↔ CLI link over a Unix
socket using length-prefixed JSON envelopes. See
[architecture.md](architecture.md#ipc-envelope).

### Kind

The type discriminator on an entry: `env_var` today, `file` reserved.
Part of the row AAD so an env-var blob can't be confused with a file
blob.

### Meta table

A SQLite table of `(key, value)` strings inside each vault.db. Holds:
schema version, vault UUID, audit-chain seed/head, entries state
hash, reserved TOTP slot.

The meta table is **unencrypted** — it's bootstrap data. The
sensitive bits live in encrypted columns of other tables.

### mlock

A syscall that pins a memory page so it never goes to swap. byn
uses it in `internal/secmem` for the master password and Argon2
workspace. Doesn't help against root, but does prevent accidental
on-disk exposure on memory pressure.

### Multi-vault

The daemon's state model: a map from vault name to per-vault entry
(store + audit logger + idle timer). Many vaults can be unlocked
simultaneously; they don't share state.

### Nonce

A unique-per-encryption number that AEAD requires. byn uses 24-byte
random nonces (XChaCha20) generated by `crypto/rand` per write.

### Pre-parser

`cmd/byn/cliscope.go:preParseGlobals` — a single linear scan over
argv that pulls out `--vault/--project/--env` regardless of position,
then returns the scrubbed argv for the subcommand FlagSet. Lets
`byn --vault X put NAME` AND `byn put --vault X NAME` both work.

### Project

The middle level of the scope hierarchy. A grouping of envs.
Example: `myapp` (with envs `dev`, `staging`, `prod`).

### Scope

Shorthand for `(vault, project, env)` — the 4-tuple that addresses an
entry. Resolution: CLI flag > env var > `.byn` > daemon default.

### Secmem

`internal/secmem` — `mlock`'d byte buffers for sensitive workspace.

### Sentinel error

A pre-declared error value matched with `errors.Is`. byn exports
sentinels like `vault.ErrLocked`, `vault.ErrNotFound`,
`vault.ErrAlreadyInit` so callers can branch on the cause without
parsing error strings.

### Slice

A self-contained chunk of work, like a sprint deliverable. byn slices are tracked on the internal roadmap and task list.

### TOFU — Trust On First Use

A trust model where the first time you see a thing, you approve it,
and subsequent uses must match what you approved. byn uses TOFU
for `.byn` files: SHA-256 of contents recorded on approval; any
content change fails the check until you re-trust.

### TUI

Text-based user interface. byn has a modal vi-style editor at
`cmd/byn/cmd_tui.go`. Slice 7 redesigns it with a left-rail
navigator for vault/project/env/files.

### Vault

The top-level container. One SQLite DB + one wrapped key + one
audit log per vault. Default name: `default`. You can have many
(`work`, `personal`, `client-X`).

### Vault key

The 32-byte random key that AEAD-encrypts every row value in a
vault. Generated at `init`, persisted only in `wrapped.key`,
unwrapped only in daemon memory while unlocked.

### Wrapping (key wrapping)

The act of encrypting a key with another key. byn wraps the vault
key with `Argon2id(password)`-derived material so the master password
is the only way back to the key.
